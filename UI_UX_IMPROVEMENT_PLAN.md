# Alert Watch UI/UX Improvement Plan
<!-- Alert Watch의 알림 성공률과 조작 신뢰도를 높이기 위한 UI/UX 개선 계획서 -->

## 목표

Alert Watch의 핵심 가치는 “사용자가 재입고 또는 구매 가능 상태를 놓치지 않도록 믿고 맡길 수 있는 것”입니다. 따라서 이번 개선의 최우선 목표는 단순한 시각 개선이 아니라, 사용자가 알림 수신 준비를 완료했는지 명확히 알고, 감시가 실제로 동작 중인지 신뢰하며, 설정을 실수 없이 마칠 수 있게 만드는 것입니다.

## 현재 진단

현재 인터페이스는 전반적인 브랜드 톤과 상태 중심 화면은 잘 잡혀 있지만, 핵심 행동 흐름에 몇 가지 단절이 있습니다.

- 브라우저 푸시 알림 켜기 버튼이 `이력 & 알림` 탭 안에 있어 최초 사용자가 놓치기 쉽습니다.
- `설정 저장 및 시작`을 눌러도 이 기기가 알림 수신 가능한 상태인지 즉시 이해하기 어렵습니다.
- 설정 폼의 URL, 키워드, 옵션명, 조건, 주기, 반복 정책이 한 화면에 같은 무게로 노출되어 초보 사용자에게 부담이 큽니다.
- 감시 상태는 펄스 점과 마지막 확인 시간에 의존하고 있어, “정상 동작 중”, “설정 필요”, “알림 미연동”, “최근 확인 실패” 같은 운영 상태가 충분히 구분되지 않습니다.
- 작동 테스트 결과가 텍스트 중심이라 성공, 경고, 실패의 원인과 다음 행동을 빠르게 파악하기 어렵습니다.
- 모바일에서 2열 폼이 유지될 수 있어 긴 한글 라벨과 버튼 텍스트가 답답하게 보일 위험이 있습니다.

## 개선 원칙

1. 알림 수신 가능 상태를 첫 화면에서 끝까지 책임집니다.
2. 사용자가 가장 먼저 해야 하는 행동을 숨기지 않습니다.
3. 고급 설정은 필요할 때만 드러내고, 기본 흐름은 짧게 유지합니다.
4. 애니메이션은 장식보다 상태 이해를 돕는 용도로만 사용합니다.
5. 카드 중첩과 과한 글래스모피즘을 줄이고, 정보 위계와 대비를 우선합니다.
6. 모바일에서도 한 손으로 설정을 완료할 수 있게 폼과 버튼을 안정적으로 배치합니다.

## 1순위 개선안. 알림 수신 온보딩 강화

### 문제

Alert Watch는 푸시 알림이 핵심인데, 현재는 알림 설정이 세 번째 탭에 숨어 있습니다. 사용자는 감시를 시작했다고 생각하지만 실제 기기 알림은 받지 못할 수 있습니다.

### 개선 내용

- `현황` 탭 상단에 알림 상태 배너를 추가합니다.
- 상태별 문구와 액션을 분리합니다.
  - 미연동: “이 기기는 아직 알림을 받지 않습니다.”, `[알림 켜기]`.
  - 권한 거부: “브라우저에서 알림 권한이 차단되어 있습니다.”, `[설정 방법 보기]`.
  - 미지원: “이 브라우저는 푸시 알림을 지원하지 않습니다.”.
  - 연동 완료: 배너는 숨기거나 작은 “알림 연동 완료” 상태로 축소합니다.
- `설정 저장 및 시작` 또는 `감시 시작` 클릭 시 알림이 꺼져 있으면 저장/시작 결과와 함께 인라인 안내를 표시합니다.
- 강제 모달은 기본값으로 사용하지 않습니다. 브라우저 권한 요청은 사용자가 `[알림 켜기]`를 명확히 누른 순간에 실행합니다.

### 구현 범위

- `public/index.html`: 현황 탭 상단에 알림 준비 배너 마크업 추가.
- `public/index.css`: 상태 배너, CTA, 권한 상태별 스타일 추가.
- `public/app.js`: `Notification.permission`, `currentSubscription`, 서비스 워커 지원 여부를 기준으로 배너 상태 업데이트.

### 완료 기준

- 첫 접속자가 5초 안에 “알림을 켜야 한다”는 사실을 알 수 있습니다.
- 감시 시작 후에도 알림 미연동 상태라면 사용자가 그 사실을 명확히 볼 수 있습니다.
- 알림 권한 거부와 단순 미연동 상태가 서로 다른 문구로 표시됩니다.

## 2순위 개선안. 설정 폼 정보 위계 재구성

### 문제

모든 입력이 같은 수준으로 나열되어 있어 초보 사용자에게 어렵게 느껴집니다. 특히 옵션명과 감지 조건은 설명이 필요한데 현재는 기본 입력처럼 보입니다.

### 개선 내용

- 설정 폼을 네 개의 논리 섹션으로 재배치합니다.
  - 기본 설정: 상품 URL, 품절 또는 구매 가능 문구.
  - 감지 조건: “문구가 사라지면 알림”, “문구가 나타나면 알림”.
  - 알림 정책: 확인 주기, 반복 횟수, 반복 간격.
  - 고급 설정: 옵션명 또는 CSS 선택자성 입력.
- 고급 설정은 기본적으로 접힌 상태로 둡니다.
- 섹션은 과한 카드 중첩 대신 `fieldset`, 얇은 구분선, 작은 섹션 제목, 간격으로 구분합니다.
- 반복 횟수가 `1`이면 반복 간격 입력은 비활성화하고 설명을 줄입니다.

### 구현 범위

- `public/index.html`: 폼 섹션 구조 재배치, 고급 설정 `details` 적용.
- `public/index.css`: 섹션 제목, 구분선, 폼 간격, 비활성 입력 스타일 정리.
- `public/app.js`: 기존 입력 ID를 유지해 저장 로직과 호환되도록 하고, 반복 간격 활성화 로직을 유지 또는 보완.

### 완료 기준

- 기본 설정만 보면 사용자가 무엇을 입력해야 하는지 바로 이해할 수 있습니다.
- 고급 설정을 열지 않아도 일반적인 감시 설정을 완료할 수 있습니다.
- 기존 저장 API 요청 구조가 깨지지 않습니다.

## 3순위 개선안. 감시 상태 신뢰도 강화

### 문제

현재 상태 피드백은 펄스 점, 마지막 확인 시간, 상태 배지 중심입니다. 하지만 “다음 확인까지 남은 시간”, “최근 확인 성공 여부”, “설정 저장 전 초안 상태” 같은 맥락이 부족합니다.

### 개선 내용

- 현황 탭에 다음 상태를 분명히 구분합니다.
  - 설정 필요.
  - 감시 중.
  - 감시 중지.
  - 알림 미연동.
  - 최근 확인 실패.
  - 구매 가능 감지.
- 마지막 확인 시간 옆에 “다음 확인까지 N초” 카운트다운을 추가합니다.
- 4초 자동 새로고침마다 장식용 프로그레스바를 넣기보다, 실제 갱신 시점에만 작은 상태 텍스트 또는 배지 변화를 줍니다.
- 사용자가 URL을 새로 입력해 저장 전 초안 상태일 때는 기존 상품 상태를 그대로 보여주지 않고 “새 설정 저장 필요” 상태를 명확히 표시합니다.

### 구현 범위

- `public/index.html`: 상태 보조 텍스트 또는 보조 배지 영역 추가.
- `public/index.css`: 상태별 배지 색상과 레이아웃 보완.
- `public/app.js`: `data.isMonitoring`, `lastCheckTime`, `hasUnsavedChanges`, API 오류를 조합한 상태 렌더링 함수 추가.

### 완료 기준

- 사용자가 현재 감시가 켜져 있는지, 알림 수신이 준비됐는지, 마지막 확인이 언제였는지 한 화면에서 알 수 있습니다.
- 서버 통신 실패가 조용히 묻히지 않고 명확히 표시됩니다.

## 4순위 개선안. 작동 테스트 결과 카드화

### 문제

현재 작동 테스트 결과는 텍스트 박스 중심이라 진단 결과를 해석해야 합니다.

### 개선 내용

- 테스트 결과를 카드형 패널로 표시합니다.
- 결과 패널에는 다음 정보를 분리해 보여줍니다.
  - URL 접근 결과.
  - 감시 문구 발견 여부.
  - 현재 판단.
  - 실제 알림 발생 조건.
  - 권장 다음 행동.
- 성공, 경고, 실패 상태마다 아이콘과 색상을 다르게 적용합니다.
- 테스트 성공 후 `[이 설정으로 저장 및 시작]` 액션을 더 명확히 연결합니다.

### 구현 범위

- `public/index.html`: 기존 `diagnostic-result` 영역을 확장 가능한 결과 카드 컨테이너로 사용.
- `public/index.css`: 진단 카드, 결과 행, 상태 아이콘 스타일 추가.
- `public/app.js`: `/api/check-site` 응답을 구조화해서 렌더링. 서버 응답이 텍스트만 있어도 우선 호환되도록 처리.

### 완료 기준

- 사용자가 테스트 결과를 읽고 다음 행동을 3초 안에 결정할 수 있습니다.
- 실패 시 원인과 수정할 입력칸이 명확히 드러납니다.

## 5순위 개선안. 모바일 및 접근성 보완

### 문제

긴 한글 라벨과 버튼 텍스트, 2열 폼, 작은 터치 영역은 모바일에서 조작성을 떨어뜨릴 수 있습니다.

### 개선 내용

- `600px` 이하에서 `.form-row-2`를 1열로 전환합니다.
- 탭 버튼 텍스트가 좁은 화면에서 줄바꿈되더라도 높이가 안정적으로 유지되게 합니다.
- 버튼 active 상태를 추가해 터치 반응을 명확히 합니다.
- `prefers-reduced-motion` 대응으로 펄스, 배경 blob, 버튼 bounce 애니메이션을 줄입니다.
- 숨김 제목은 `display:none` 대신 스크린리더 전용 클래스로 바꿔 접근성을 보완합니다.

### 구현 범위

- `public/index.html`: 숨김 제목에 `.sr-only` 클래스 적용.
- `public/index.css`: 모바일 1열 폼, 터치 영역, reduced motion, sr-only 스타일 추가.

### 완료 기준

- 모바일에서 입력 폼과 버튼 텍스트가 겹치거나 잘리지 않습니다.
- 모션 감소 설정을 켠 사용자는 반복 애니메이션을 보지 않습니다.
- 탭과 주요 상태 제목이 보조 기술에서도 의미 있게 읽힙니다.

## 6순위 개선안. 시각 스타일 정리

### 문제

현재 UI는 어두운 글래스 패널과 발광 효과가 많아 제품 신뢰감보다는 장식감이 앞설 수 있습니다.

### 개선 내용

- 글래스모피즘은 메인 콘솔과 주요 상태 영역에만 유지합니다.
- 폼 내부 섹션은 더 낮은 투명도, 더 명확한 경계, 충분한 대비를 사용합니다.
- 구매 가능 상태, 오류 상태, 알림 미연동 상태의 색상 의미를 일관되게 정리합니다.
- `direct-buy-link`의 무한 bounce는 구매 가능 감지 직후 짧은 시간만 강조하거나, 반복 움직임을 줄입니다.

### 구현 범위

- `public/index.css`: 색상 토큰, 상태 배지, hover, active, focus 스타일 정리.
- `public/app.js`: 필요 시 구매 가능 감지 직후에만 강조 클래스를 붙이는 방식으로 조정.

### 완료 기준

- 화면이 더 차분하고 운영 도구처럼 신뢰감 있게 느껴집니다.
- 중요한 액션과 상태만 시각적으로 강하게 보입니다.

## 권장 구현 순서

1. 알림 상태 배너와 `app.js` 상태 동기화.
2. 저장 및 감시 시작 흐름에서 알림 미연동 안내 연결.
3. 설정 폼 섹션화와 고급 설정 접기.
4. 모바일 1열 폼과 접근성 보완.
5. 작동 테스트 결과 카드화.
6. 감시 상태 보조 정보와 시각 스타일 정리.

## 변경 파일 계획

- `public/index.html`.
  - 알림 준비 배너 추가.
  - 설정 폼 섹션 구조 재배치.
  - 진단 결과 영역 확장.
  - 숨김 제목 접근성 방식 개선.
- `public/index.css`.
  - 알림 배너, 폼 섹션, 진단 카드, 모바일 1열, reduced motion 스타일 추가.
  - 과한 반복 애니메이션과 카드 중첩 스타일 정리.
- `public/app.js`.
  - 알림 권한 및 구독 상태 렌더링.
  - 저장/시작 후 알림 미연동 안내.
  - 진단 결과 구조화 렌더링.
  - 상태 신뢰도 보조 표시.

## 검증 계획

### 수동 검증

- 신규 브라우저 프로필에서 최초 접속 후 알림 배너가 보이는지 확인합니다.
- 알림 권한 허용, 거부, 기본값 상태별 문구와 버튼 동작을 확인합니다.
- 알림 미연동 상태에서 `설정 저장 및 시작`을 눌렀을 때 사용자가 놓치지 않는 안내가 나오는지 확인합니다.
- 상품 URL과 키워드만 입력해 기본 감시를 저장할 수 있는지 확인합니다.
- 고급 설정을 열지 않아도 기존 API 요청이 정상 동작하는지 확인합니다.
- 작동 테스트 성공, 경고, 실패 상태가 서로 명확히 구분되는지 확인합니다.
- 모바일 폭에서 폼, 탭, 버튼 텍스트가 겹치거나 잘리지 않는지 확인합니다.

### 기술 검증

- `npm run build` 또는 프로젝트에서 제공하는 빌드 명령을 실행합니다.
- 로컬 서버에서 데스크톱과 모바일 뷰포트 화면을 확인합니다.
- 브라우저 콘솔에 JS 오류가 없는지 확인합니다.
- 서비스 워커 등록과 푸시 구독 해제/재구독 흐름이 깨지지 않는지 확인합니다.

## 성공 기준

이번 개선이 성공했다면 사용자는 다음 질문에 망설임 없이 답할 수 있어야 합니다.
